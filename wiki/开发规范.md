# 开发规范

本页约束代码改动、发布动作和文档同步规则。

## 基本原则

1. 数据正确优先。
2. 权限边界清晰。
3. 前后端链路一致。
4. React 迁移按 typed API、Query、UI state、view 的单向依赖执行。
5. 新依赖需要对应明确性能或契约价值，并写入 `package.json`。

## 改动入口

| 领域 | 主要落点 |
| --- | --- |
| React 页面与交互 | `src/react/`、`src/styles.css` |
| 前端接口与状态 | `src/react/api/`、`src/react/state/`、`src/domain/` |
| 前端入口与版本 | `src/main.tsx`、`src/react/version.ts`、`src/react/releaseNotes.ts` |
| 后端接口与权限 | `server.py`、`server_app/http.py` |
| 后端读写与领域逻辑 | `server_app/repositories/`、`server_app/services/` |
| 文档 | `wiki/`、`README.md` |

## 必做检查

```bash
npm run check
```

生产构建和浏览器回归：

```bash
npm run build
npm run test:e2e
npm run benchmark
```

涉及接口联动、权限、分页、缓存、导出、流程推进时，再补：

```bash
npm run smoke:api
```

## 业务约束

- IACUC 是贯穿笼位、数量统计表和结算链路的核心业务键
- `项目来源` 是 `funding` 的 source of truth
- “已取材”和“设为空”保留不同业务语义
- 数量统计表和动态笼位图是并存数据入口
- 结算流程是月度结算单流程

## 禁止事项

- 不手工改 `dist/` 打包产物
- 不直接修改 `data/` 下的数据库和运行数据文件
- 不跳过 `npm run check` 直接提交或发布
- 不把前端显示正确当成后端链路正确
- 不复用旧 tag 修补新内容

## 文档同步规则

- 接口、数据结构、部署方式、用户操作发生变化时，同步更新对应 `wiki/` 页面
- `wiki/` 是正式文档源
- `docs/` 是分析、契约和迁移文档源

## 前端配色规则

- 全局颜色入口集中在 `src/styles.css` 的 `:root`
- 业务状态优先使用语义变量，例如 `--selection-*`、`--reserved-*`、`--warning-*`、`--danger-*`
- 表格选中态、批量选择和当前对象统一使用 `--selection-*`
- 笼位“已预约”和待进驻预留统一使用 `--reserved-*`
- 扫码盘点相关状态统一预留 `--inspect-*`
- 详细规则见 `docs/contracts/ui-color-system.md`

## 保存链路性能规则

- 高频保存接口优先使用 repository 局部写入，避免通过 `write_state()` 重写整库
- 写入接口返回最新对象和受影响对象，前端先增量合并并提示成功
- 保存后的刷新限定在当前房间、当前列表页、当前详情或当前数量统计表
- 新增写路径需要输出 `[perf]` 日志或 `perf` 响应摘要，便于定位慢点
- 列表查询新增筛选或排序列时，需要补索引并使用 `EXPLAIN QUERY PLAN` 检查
- 性能基准只使用系统临时目录，不直接写入 `data/`
