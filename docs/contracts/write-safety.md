# 业务写入安全约束

## 写入意图

- 创建操作使用 `POST`，客户端为每次新草稿生成新的 ID。
- 编辑操作仅从已保存记录进入，使用 `PUT /{id}`。
- 连续录入页面在创建成功后立即回到新草稿状态。
- 状态推进使用专用动作接口，服务端在单个事务内校验当前状态。

## 并发版本

- 已保存的业务记录通过 `updatedAt` 表示服务端版本。
- 前端更新请求携带 `expectedUpdatedAt`。
- 服务端在事务内比对当前版本；版本变化返回 `409` 和“刷新后重新编辑”提示。
- 服务端负责写入新的 `updatedAt`，客户端不生成记录版本。

## 审计与恢复

- 关键写入继续记录操作者、时间、变更前后快照和业务实体 ID。
- 恢复脚本只通过业务写入服务创建新记录，恢复记录携带来源审计事件 ID。
- 恢复后的笼卡使用新的批次 ID 和二维码，避免与历史记录冲突。

## 当前覆盖

第一批覆盖待接收批次、数量统计表、笼位占用和项目负责人计费配置。流程状态推进、导入和结算生成通过各自的事务与状态校验保护。基础设施、账号和报销台账进入后续统一版本校验批次。
