# React/Vite 性能升级任务分解

## 全局约束

- 新模块职责单一，依赖方向固定为 view -> state/query -> api -> server。
- API 和数据库变更保持序列化契约兼容。
- 业务结果、权限、视觉和操作流程与 `v0.5.25` 等价。
- 每项任务完成后运行 S.U.P.E.R 检查和对应测试。

## Phase 1：基线与构建

| 任务 | 优先级 | 工作量 | 依赖 | 验收 |
|:-----|:-------|:-------|:-----|:-----|
| 1.1 建立性能临时数据和基准脚本 | P0 | M | - | 可生成 1 万笼位/10 万记录并输出 P50/P95 |
| 1.2 搭建 React/Vite/TypeScript 和测试 | P0 | M | - | build、check、Vitest 可运行 |
| 1.3 建立 typed API、Query 和 UI state | P0 | L | 1.2 | 登录和 bootstrap 契约可测试 |

## Phase 2：业务视图迁移

| 任务 | 优先级 | 工作量 | 依赖 | 验收 |
|:-----|:-------|:-------|:-----|:-----|
| 2.1 应用壳、登录、主页 | P0 | L | 1.3 | 登录后可操作 `<1.5s` |
| 2.2 笼位、笼卡和扫码 | P0 | XL | 2.1 | 预留、入驻、打印、接收和扫码通过 |
| 2.3 饲养费、数量统计和结算 | P0 | XL | 2.1 | 保存、预览、导出和流程创建通过 |
| 2.4 流程中心和系统设置 | P1 | L | 2.1 | 权限、详情、推进和管理功能通过 |

## Phase 3：服务与数据性能

| 任务 | 优先级 | 工作量 | 依赖 | 验收 |
|:-----|:-------|:-------|:-----|:-----|
| 3.1 笼卡热字段迁移和回填 | P0 | M | - | 新旧库行为一致 |
| 3.2 复合索引和查询计划 | P0 | M | 3.1 | 目标列表 API P95 `<300ms` |
| 3.3 静态缓存、gzip、ETag、Server-Timing | P1 | M | 1.2 | 哈希资源长期缓存且响应计时可见 |

## Phase 4：切换与发布

| 任务 | 优先级 | 工作量 | 依赖 | 验收 |
|:-----|:-------|:-------|:-----|:-----|
| 4.1 切换入口并删除旧运行模式 | P0 | L | Phase 2 | 正式入口只加载 React 构建 |
| 4.2 更新 Docker、离线包和发布脚本 | P0 | M | 4.1, 3.3 | 镜像和离线包无需目标机 Node |
| 4.3 完整回归、性能验收和 Wiki | P0 | L | 全部 | check、build、E2E、基准通过 |

## 并行策略

- Phase 1 的基准脚本和前端构建可并行，文件重叠低。
- Phase 2 各视图共享 typed state，完成应用壳后按业务域并行；最终统一回归。
- Phase 3 可与业务视图迁移并行，数据库迁移和前端文件无重叠。
